MVC’s massive

  • MVC 的介绍

    • Model: 负责数据的处理
    • View: 负责视图的显示, 以及事件的监听
    • Controller: 用于流程控制

  • Massive 的 原因 / ViewController 需要做的事情

    • 表示层逻辑
    • 负责 View 的布局, 加载, 展示顺序
    • 各种事件的响应
    • 网络请求
    • …

Controller 的臃肿 会给 阅读 维护 带来困难
对新接手编写代码的同学没有约束, 他们可以肆意地将表示逻辑代码扔在Controller 的各个位置

对 MVVM 的了解


ViewModel: 会从数据源(网络请求/数据库/…)中获取原始数据,然后处理解释和转换的表示逻辑,变成 view 能够直接呈现的数据. 暴露给 controller 必要的视图工作信息

ViewModel 与 Controller 之间采用 RAC(ReactiveCocoa) 进行统一的数据信号绑定

解决了什么问题:
Controller 可以减少的工作: 网络请求, 数据转换的表示逻辑, 数据存储
为VC 瘦身, 方便阅读以及维护

但同时, RAC 学习成本较高,而且使用的时候调试有点麻烦

采用 MVVM 的思想

在计算机领域有一句话叫做: 没什么是加一个中间层解决不了的问题

  • ViewModel: 对数据的解释和转换进行处理, 输出可与 view 直接交互的数据, 采用传统的 UI 事件监听方式, 而不使用RAC的通知机制或者双向绑定.
  • DataController: 接手数据源的处理, 拿到原始数据后, JSON/XML -> Model, 再加数据加工成 ViewController 需要的数据形式 ,相当于一个数据接收器,解析器.

应用

Controller

总结

优点

  • 层次清晰: 设计模式上应该尽量满足: Single Function, Single Task, 同样, 每个文件也应有其对应的任务.
  • 重构成本低: 重写代码并不多, 只是较多的粘合代码, 但达到了较好的复用的效果 -> ”Don’t repeat yourself”
    缺点
  • 文件太多: 会影响启动速度, 还有包的大小
  • 交互逻辑变多: 在 DataController— Controller — ViewModel 间 数据来回传递,多了很多粘合的代码.

什么是缓存

  • 读写速度,存储容量介于CPU与主存之间的存储器,用于缓存一些临时文件
  • 读写速度: 寄存器 > 缓存 > 主存
  • 存储容量: 主存 > 缓存 > 寄存器
    Q: 为什么需要缓存/使用缓存的场景
    A: 提高软件的响应速度,节约流量,提高用户体验,一个用户量高的软件必定离不开缓存的使用

YYCache

目录

  • YYCache 主要分为内存缓存 YYMemoryCache 与磁盘缓存 YYDiskCache
  • YYCache类是封装好的,主要向外提供的接口
  • 内存缓存提供容量小但高速的存取功能

  • 磁盘缓存提供大容量但低速的持久化存储

  • 存储位置 Library/Cache

  • 查: 先看memory中有没有, 没有的话再到disk中查找,如果没有返回nil
  • 存: 先往memory里面存,存往再向disk中存
  • 取: 先在memory中拿,没有的话再向disk中拿,拿得到的话再向memory存一遍(以便于下次取的时候更快),拿不到便返回nil
  • 删: 先从memory删除,再从disk中删

    如何设计一个缓存库

  • 具有内存缓存以及磁盘缓存
  • 线程安全(锁的挑选以及使用)
  • 缓存替换策略的选择(Optimal,FIFO,LRU…)

思考

Q: 开发过程中什么样子的数据需要备份
A: 1.需要可备份存储的资料 -> Document

- 微信聊天记录
- 单机游戏进度

2. 不需要可备份存储的资料 -> `Library/Cache`
- 朋友圈,新闻等 资讯类

Q: App设置中清理缓存 清理的是哪个文件下的资源
A: 看清理缓存 对应的 清理机制 一般清理的是Library/Cache

Q: 存放在 document, Library, tmp 外的文件夹 系统如何对待
A: 系统只备份document 跟 Library/preference 下的数据 , 且重启手机,系统也只删除tmp下的数据

Q: 为什么yycache里面要使用锁
A: 在进行读写操作修改链表的时候,不可以有并发操作同时修改链表
eg: 售货员有1张票 当有人买了 进行修改删除的时候 其他人也可以进行操作 便会出现错误.

自动化批量打包

Veröffentlicht am 2017-08-27

背景

手动打包是一个机械 而又 非常浪费时间的过程,每当测试人员想你要测试包的时候 , 你就不得不重复这一次次过程.从我自己现在工作中的一个项目的场景谈起,他最终会需要生成n个包(大于十个且数量随销售增加),但它使用的是同一套源代码,只是需要修改plist中的一些参数,eg:`dispalyName/Icon/…如果是手动打包的话,每次更新版本就是你听着风扇狂响崩溃的时候

相关知识

1.xcodebuild -list [[-project <projectname>]|[-workspace <workspacename>]] [-json] -> 先查看一下项目的scheme & configrations:

2.build 生成 .app文件

  • <buildsetting>=<value>: 一般设置证书(CODE_SIGN_IDENTITY),配置文件(PROVISIONING_PROFILE),SYMROOT
  • <buildaction>: 默认是build, 其他 buildaction如下

3.xcrun:导出成包

  • <#.ipaPath/xxx.ipa#>: .ipa包导出的路径,和.ipa包的名字

批量打包思路

批量打包的大体过程为两个步骤
1.代码编译成.app文件
2..修改内容,然后重签名 生成.ipa包

我们在打包的时候浪费最多时间的就是第一个步骤编译生成.app文件,根据以上所说的项目背景,我们应该在codesign上下功夫,也就是说 但当编译成.app文件后,通过修改.plist文件再重签名达到自动化批量打包的效果

1.codesign: 重签名

codesign -f -s "iPhone Distribution: xx co., LTD" --entitlements <#EntitlementsPath#> $ipaPath/Payload/xxx.app

  • entitlements: 自己本地创建的授权文件.可以参照Inside Code Signing中的设置,如下

2.查看签名信息:codesign -vv -d Example.app

查看Authority 开头的三个中代表:我的证书,Apple私钥签名我的公钥,公证处(CA)

3.检测重签名是否成功

codesign --verify Example.app: -> 没有任何输出则代表签名完好
